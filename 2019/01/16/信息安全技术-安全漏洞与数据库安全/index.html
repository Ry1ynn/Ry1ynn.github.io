发表于 2019-01-16 | 分类于 NISP二级

1.1-安全漏洞的产生与发展

安全漏洞的基本概念

  • 何为漏洞？

    也称脆弱性，计算机系统天生的类似基因的缺陷，在使用和发展过程中产生意想不到的问题（冯诺伊曼

  • 漏洞的含义：

    • 漏洞本身随着信息技术的发展而具有不同的含义和范畴
    • 基于访问控制的定义，逐步发展到，设计系统安全流程、设计、实施、内部控制等全过程的定义

  • 学者们对漏洞的定义

    • 从访问控制角度定义
    • 从风险管理角度的定义
    • 使用状态空间描述的方法定义

  • 标准机构的定义

    • 存在于评估对象中，违反安全功能要求的弱点
    • 威胁源可以攻击或触发的信息系统、系统安全流程、内部控制或实施中的弱点
    • 威胁可以利用的一个或一组资产的弱点；是评估对象TOE中的弱点；是在信息系统或其环境的设计及实施中的缺陷、弱点或特性

  • 漏洞的理解

    • 从生命周期的角度出发
    • 信息技术、产品、系统在需求、设计、实现、配置、维护和使用等过程中
    • 有意或无意产生的缺陷
    • 这些缺陷一旦被恶意主体所利用
    • 就会造成对信息产品或系统的安全损害
    • 从而影响构建于信息产品或系统之上正常服务的运行
    • 危害信息产品或系统的安全属性

  • 安全缺陷：是信息系统或产品自身“与生俱来”的特征，是其的固有成分

漏洞产生的原因

  • 技术原因：
    • 软件系统复杂性提高，质量难以控制，安全性降低
    • 公用模版的使用引发了安全问题
  • 经济原因
    • 柠檬市场效应（没人再愿意花高成本造高质量产品
  • 环境原因
    • 从传统的封闭、静态和可控变为开放、动态和难控
    • 攻易守难

漏洞的分类

  • 分类的目的
    • 准确的区分和描述不同的漏洞
    • 有助于漏洞的发布、存储和查询
  • 分类实例：
    • NVD：代码注入等21种类型
    • CNNVD：根据漏洞形成分成22种类型

1.2-安全漏洞的发现与修复

  • 漏洞的发现
    • 静态漏洞挖掘
    • 动态漏洞挖掘

静态漏洞挖掘

  • 不运行代码而直接对代码进行漏洞挖掘的方法
  • 适用对象
    • 完整的或不完整的源代码
    • 二进制代码
    • 中间代码片段
  • 方法
    • 流分析
    • 符号执行
    • 模型检测

动态漏洞挖掘

  • 在代码运行的状态下，通过检测代码的运行状态，或根据测试用例结果来挖掘漏洞的方法
  • 特点：与静态相比，动态分析方法的优势在于其分析结果的精确，即误报率低
  • 方法
    • 模糊测试
    • 动态污染传播

安全漏洞的修复

  • 安装补丁是主要技术手段之一
  • 补丁修复相关问题
    • 打什么样的补丁？补丁质量问题
    • 如何打补丁？操作方式问题
    • 什么时间打补丁？修复时机问题
  • 补丁分类
    • 文件类型：源代码形式、二进制形式
    • 内存角度：文件补丁（冷补丁）、内存补丁（热补丁）
  • 补丁安装时要注意的问题
    • 补丁安装部署之前需要经过必要的测试
    • 需要从可靠来源不断获取最新补丁信息
    • 安装补丁时，要做好备份和相应的应急措施
  • 标准化的安全配置：美军率先推行标准化的安全配置与核查IAVA
  • 根据漏洞分析和风险评估的安全加固：传统的安全加固手段越来越难以应付日益复杂的攻击行为，漏洞信息的及时披露和分发越来越重要
  • 加固核查与问责：通过安全审计核实漏洞消除情况和效果。

1.4-数据库安全

数据库系统基本概念

  • 数据库：长期存储在计算机内的、有组织的、可共享的数据集合
  • DBMS：
    • 一种操纵和管理数据库的大型软件
    • 用于建立、使用、维护数据库
    • 对数据库进行统一的管理和控制
    • 以保证数据库的安全性和完整性
  • 用户请求——DBMS——数据库（类似：用户请求——图书管理员——图书馆）
  • 数据库系统关系
    • 数据库—操作系统—DBMS—应用开发工具—应用系统
    • 数据库和DBMS：数据库管理员
    • 应用开发工具：程序员
    • 应用系统：用户

关系型数据库系统特点

  • 数据结构化和共享性
    • 库、表、字段
    • 结构化查询语言SQL
  • 数据独立性：物理独立性、逻辑独立性
  • 数据的完整性约束
    • 实体完整性（通过主键约束条件
    • 参照完整性（通过主外键约束条件
    • 用户自定义完整性（通过CHECK子句
  • 数据由DBMS统一管理和控制：查询引擎、事务控制、数据存储模块

结构化查询语句

  • SQL语句格式
    • 基本语句约30条，每条语句都请求DBMS完成一个动作
    • 如：delete from table_name where column_name < 1
      • delete：动词
      • from：关键字
      • 有表名、列名
      • where引导子句
      • where后跟表达式
    • SQL语句分类
      • 数据定义：create
      • 数据查询：select
      • 数据操纵：update
      • 数据控制：grant
      • 事务控制：begin
      • 嵌入式/服务端编程：open

数据库安全

  • 保密性：数据泄漏
  • 完整性：执行正确的修改操作
  • 可用性：数据库死锁，性能下降或无法访问

数据库安全需求

  • 物理数据库完整性
  • 逻辑数据库完整性
  • 数据单元安全性
  • 可审计
  • 访问控制

数据库安全措施

  • 用户标识与鉴别
    • 用户鉴别
      • 操作系统
      • 数据库
      • 第三方
    • 用户管理策略
      • 默认账号管理
      • 账号锁定策略
  • 授权与访问控制
    • 设置用户对不同的数据对象允许的操作权限
      • 系统权限：数据库实例级或数据库级
      • 模式权限：数据库对象：表、视图
      • 数据对象权限：数据对象：记录级
    • 资源控制
      • CPU资源
      • 会话资源
        • 每用户session
        • 连接超时
        • 死锁控制
      • 存储资源
        • 护具存储空间限制
        • 临时存储空间限制
  • 数据加密
    • 数据加密方式
      • 传输加密
      • 存储加密
    • 数据存储加密层级
      • os层加密
      • DBMS内核层加密
      • DBMS外层加密
  • 安全审计
    • 数据库审计（也称标准审计
      • 用户级审计：安全审计员设置，用户行为审计
      • 系统级审计：管理员设置，用户授权或系统权限操作审计
    • 细粒度审计
      • 监视对数据对象的操作

数据库完整性

  • 完整性保证：完整性约束
  • 完整功能性约束条件
    • 实体完整性（primary key定义，唯一，不为空
    • 域完整性（表定义中的check、foreign key约束，default、not null定义
    • 参照完整性（foreign key定义，关联
    • 用户定义完整性（生成时定义，非空，唯一，布尔

数据备份与恢复

  • 备份：数据冗余技术
    • 数据转储
    • 日志文件
  • 恢复：从冗余数据中实施数据库恢复
    • 事务故障的恢复（事务处理
    • 系统故障的恢复（事务处理
    • 介质故障的恢复（通过转储数据恢复

数据库运行安全防护

  • 数据库面临的安全威胁：特权滥用、平台级数据库漏洞、SQL注入、拒绝服务攻击、不健全的审计和认证、数据泄漏、数据库通信协议漏洞

  • 层层设防：防护体系建设

    • SQL请求—用户标识与鉴别–DBMS存取控制–数据加密–审计追踪
    • 对应于：各种应用–安全边界–查询引擎—DBMS选件–事务引擎

  • 检查、监控、审计（事前、中、后

    • 事前检测：安全特性检测

      • 安全配置：补丁、协议
      • 账号：用户名密码、口令策略、权限
      • 存储过程
      • 触发器
      • 备份
    • 事中运行监控
    • 事后审计
      • 审计对象（对谁进行审计
        • 标准审计（系统级、用户级
        • 细粒度审计（对象级
      • 审计内容（对什么行为进行审计
        • 访问数据库应用程序、位置及用户信息
        • 用户操作的数据、操作是否成功
